Thought for 24s
You want something between raw COR24 assembler and PL/SW: a macro-assembler that keeps assembly close to the machine, but adds enough abstraction to improve structure, reuse, portability, and readability.

What to optimize for
For your use case, the sweet spot is usually:

Real macros with parameters, defaults, local symbols, recursion/iteration

Conditional assembly for target variants, debug/release, feature flags

Structured control-flow macros so open-coded branches are less error-prone

Sections / relocation / symbol visibility so larger programs can be organized cleanly

Good listing / cross-reference / diagnostics so it still feels like assembly, not a mysterious preprocessor

Libraries of reusable macros for calling conventions, prolog/epilog, MMIO, tables, dispatch, loops

That is exactly the territory where IBM’s HLASM became strong: not just “MACRO/MEND,” but a whole ecosystem of conditional assembly, reusable macro libraries, and structured-programming macro sets like IF / ELSEIF / ELSE / ENDIF, DO / ENDDO, CASE, SEARCH, and SELECT. IBM explicitly documents these structured-programming macros as part of the Toolkit Feature, intended to simplify complex control flow and reduce branch mistakes. 

My recommendation
For COR24, I would not try to copy one historical macro-assembler wholesale. I would build a hybrid macro-assembler inspired by the best ideas from several systems.

Best reference model: IBM HLASM style
This is the best model if your goal is “portable high-level assembly” rather than just textual substitution. HLASM’s structured macro facilities and strong conditional-assembly support are the closest historical match to what you described. IBM’s materials describe HLASM and the Toolkit Feature as providing structured constructs and broader tools for understanding and maintaining assembler applications. 

Best second model: DEC MACRO-11 / VAX MACRO
DEC’s macro assemblers are a very strong reference for practical macro power without going too far from the metal. DEC documentation highlights features like:

user-defined macros

conditional assembly

program sectioning

relocatable object modules

global symbols

system macro libraries

listing and cross-reference support 

That combination is very attractive for your COR24 environment, especially if you want separately assembled modules and eventual loaders/linkers.

Best “high-level syntax in assembler” reference: MASM/TASM family
MASM is worth studying because it popularized assembler-level structured directives such as .IF, .WHILE, and .REPEAT. Even the Microsoft docs still reflect these as first-class “high-level directives.” 

These are useful not because MASM itself is your ideal, but because the syntax is familiar and easy to teach.

What I would not do
Do not make it a full language

Avoid:

too much implicit state

Do not rely only on textual substitution

You want:

scoped macro locals

hygienic-ish generated labels

compile-time expressions

controlled conditionals

maybe named blocks that validate nesting

That is the difference between “assembler with macros” and “usable macro-assembler.”

Best architecture for your toolchain
Given what you already have, I would strongly consider this layering:

Option A: Preprocessor macro-assembler in front of existing as24
Flow:

Thought for 12s
You want a new, separate COR24 macro-assembler inspired primarily by IBM HLASM, but deliberately limited in scope so it does not grow into PL/SW. The goal is not “fancier text substitution.” The goal is a structured assembly layer that still exposes registers, labels, memory layout, and branch behavior, while removing the repetitive and error-prone parts.

That means the right design center is:

structured control-flow lowering

real macro expansion with scope and validation

conditional assembly

reusable include/copy libraries

diagnostics/listings good enough that expanded code is still understandable

IBM’s HLASM Toolkit Feature is exactly the historical reference for that mindset: its structured macro families include IF/ELSEIF/ELSE/ENDIF, DO/DOEXIT/ITERATE/ENDDO, SELECT/WHEN/OTHRWISE/ENDSEL, plus related search/case forms, specifically to simplify control flow and reduce branch errors. 

The boundary: what belongs here vs. PL/SW
This new macro-assembler should do
Make assembly safer and more structured

Generate labels/branches correctly

Encapsulate common assembler idioms

Support compile-time decisions

Organize source into reusable macro/include libraries

Help with prolog/epilog, loops, case dispatch, tables, register-save patterns, MMIO wrappers

It should not do
Become a typed language

Add rich runtime abstractions

Hide calling conventions too much

Grow expression semantics toward C/PL/I/PL/SW

Introduce “business logic” constructs that belong in PL/SW

That line matters, because IBM HLASM’s power comes from macro and conditional assembly language, not from turning assembler into a general-purpose high-level language. IBM’s macro language explicitly supports arithmetic/logical/character expressions and variable symbols for assembly-time control and substitution. 

What else beyond control-flow macros?
If control-flow is the anchor, I would add only the things that directly support “structured assembly” rather than “another compiler.”

1. Real macro semantics
Not mere token paste.

You want:

positional and keyword parameters

defaults

variadic operands later

local symbols per expansion

unique generated labels

compile-time variables

nested expansion

basic validation and error messages

This is core HLASM territory: local/global variable symbols, conditional assembly expressions, and substitution are part of the intended model, not an afterthought. 

2. Conditional assembly
This is the second pillar after control-flow macros.

Use it for:

debug/release

CPU/ISA variant selection

emulator-only helpers

optional tracing

compile-time feature flags

selecting alternate implementations

Without this, the macro system stays shallow. With it, you can write real macro libraries.

3. COPY / include libraries
IBM’s COPY exists exactly to reuse frequently needed source fragments from source libraries. 

For COR24, this becomes:

register/save-restore helpers

interrupt entry/exit

UART helpers

table and descriptor generators

structured programming macro packages

standard equates and MMIO layout declarations

4. Procedure/block scaffolding
Not full procedures as a language feature, but structured wrappers for common assembly patterns:

PROC / ENDP

optional USES=...

optional save/restore templates

local-label scope

named exit labels

This keeps routines readable without inventing a runtime model.

5. Small compile-time data/layout helpers
A carefully limited version is useful:

EQU

offset helpers

repeated field/table generation

maybe a very small STRUCT/record-offset facility

This should stop at layout assistance, not become a type system.

6. Listing / expansion / xref support
This is easy to underestimate. It is one of the things that makes a macro-assembler usable in practice.

You want:

source-with-expansion view

final lowered .s

macro trace on demand

symbol table

label xref

nesting diagnostics

Because once you introduce structured macros, you need a clean way to inspect what they became.

Stage 0 — Define the contract
Before code, define the product boundary.

Deliverables

Victory condition
You can explain the entire product as:

“A structured macro-and-conditional-assembly front-end for as24.”

That sentence should remain true the whole life of the project.

Stage 1 — Real macros, but no structured control yet
Build the machinery first.

Features
MACRO / MEND

defaults

macro-local symbols

nested macro calls

Why first
Because all later structured forms depend on this being solid.

Example

MEND
Victory condition
You can implement reusable helpers without label collisions, and the expansion is deterministic and debuggable.

Draw the line
Do not add loops or conditionals in the macro language yet.

Stage 2 — Conditional assembly
Now give the macro system compile-time intelligence.

Features
AIF-style or your own IFASM/ELSEASM/ENDIFASM

symbol-defined checks

integer/string-ish compile-time values

macro-time variables

feature flags

assembly-time comparisons

conditional COPY

HLASM’s macro and conditional assembly model is a strong precedent here: arithmetic, logical, and character expressions with variable symbols are first-class parts of the macro language. 

Examples

  COPY debug_helpers
ENDIFASM
IFASM ISA_LEVEL >= 2
  ...
ELSEASM
  ...
ENDIFASM
Victory condition
You can write one macro library that adapts itself to configuration flags and validates its parameters.

Draw the line
Still no general-purpose compile-time programming language. Keep it declarative and small.

Stage 3 — Structured IF
This is your first real HLASM-inspired win.

IBM documents the IF family as IF, optional ELSE, optional ELSEIF, then ENDIF. 

Features
IF

ELSEIF

ELSE

ENDIF

Design choice
Do not try to parse arbitrary boolean algebra first. Start with a constrained operand model.

For example:

IF cc_eq, r0, 0
  ...
ELSEIF cc_lt, r1, 10
  ...
ELSE
  ...
ENDIF
or:

IF ZSET
  ...
ENDIF
depending on how your ISA models condition codes.

Why not more at first
The hard part is:

nesting

label generation

correct lowering

readable diagnostics

not rich condition syntax.

Victory condition
Nested IF/ELSEIF/ELSE/ENDIF lowers correctly 100% of the time and is pleasant to debug.

Draw the line
No short-circuit boolean parser yet. No AND/OR/NOT unless it is clearly assembly-time or branch-template-based.

Stage 4 — Structured DO loops
IBM’s DO macro set includes DO, optional DOEXIT, optional ITERATE, and ENDDO. 

Features
DO

ENDDO

DOEXIT

ITERATE

Start with these loop forms
infinite loop with structured exits

while-precondition loop

until-postcondition loop

Examples:

DO
  ...
  DOEXIT cc_eq, done
  ...
ENDDO
DO WHILE, cc_ne, r0, 0
  ...
ENDDO
Why this stage matters
This removes a lot of hand-written branch spaghetti while still feeling like assembly.

Victory condition
You can write scanning loops, UART polling loops, and state-machine loops without raw backward branch labels.

Draw the line
Do not add FOR with high-level induction semantics unless it lowers trivially and visibly.

Stage 5 — SELECT / CASE
IBM’s toolkit explicitly includes SELECT, WHEN, optional OTHRWISE, ENDSEL, and also CASE-related forms. 

Features
SELECT expr

WHEN value

OTHERWISE

ENDSEL

Possibly later:

range cases

fallthrough-disabled by default

jump-table lowering only as an optimization, not a semantic requirement

Why this belongs
A lot of assembler control flow is dispatch:

opcode dispatch

command interpreters

parser state machines

MMIO command handlers

This is still structured assembly, not high-level language creep.

Victory condition
You can implement command interpreters and dispatch tables without hand-managed compare/branch ladders.

Draw the line
No pattern matching. No arbitrary expression evaluation beyond what dispatch needs.

Stage 6 — Small structural helpers
Only after the core is solid.

Candidates
PROC / ENDP

USES=...

LEAVE name

BLOCK / ENDBLOCK

lightweight STRUCT / offsets

REPT / simple assembly-time repetition

Why late
These are useful, but they are also where scope creep starts.

Victory condition
They save boilerplate without hiding machine behavior.

Draw the line
If a helper starts looking like a programming language statement rather than assembly organization, stop.

What to postpone deliberately
These are tempting, but I would put them outside the initial “victory line.”

1. Full compile-time language
No general string engine, no recursive metaprogramming playground, no elaborate token rewriting DSL.

2. High-level expression parser for runtime conditions
A limited condition grammar is enough for structured lowering. You do not need PL/I conditions inside assembler.

3. Full type/record system
Offsets and layout helpers are fine. A real type system belongs elsewhere.

4. Optimizing transformations
The macro-assembler should lower structure cleanly. It does not need to become an optimizer.

5. Relocatable object format and linker changes
Useful later, but outside the first declaration of success unless you already planned an object-format evolution.

Where to declare victory
There are really three possible finish lines.

Victory Line A — Minimal, practical success
This is the one I recommend.

At that point, you have a genuine HLASM-inspired structured macro-assembler that is clearly distinct from PL/SW and clearly more powerful than text substitution. That is the best stopping point.

Victory Line B — Nice-to-have completion
Add:

This is good polish, but not necessary to claim success.

Victory Line C — Too far
If you start adding:

rich expressions

hidden runtime semantics

typed variables

complex declarations

optimizer-like behavior

then you are drifting into “mini-PL/SW in assembler clothing.” That is where to stop.

A practical rule of thumb
When deciding whether to add a feature, ask:

“Does this primarily help me write assembly more safely and readably, or does it start helping me write high-level program logic?”
If it helps write assembly safely/readably: it belongs here.

If it helps write high-level logic: it belongs in PL/SW.

That one rule will save this project from ballooning.

Thought for 31s
Yes — that is a very good demo target.

An RPG-II compiler is a great fit because RPG-II source is highly regular, fixed-format, card-oriented, indicator-heavy, and dispatch-heavy. That means a macro-assembler can visibly help with exactly the things it should be good at:

structured control flow

compile-time table generation

repetitive decoder/dispatcher patterns

reusable prolog/epilog and scanner helpers

clearer handling of indicator-conditioned logic

Historically, RPG programs are organized around specification types such as F, E, I, C, O, with calculation specifications describing calculations and their order, and output specifications describing output records and fields. IBM’s documentation explicitly describes calculation specs as defining the order of calculations and notes that they can also control some input/output operations. 

That gives you a natural path for demos: do not try to compile all of RPG-II first. Instead, build a sequence of small compiler slices where each slice showcases one macro-assembler capability.

3. What the compiler actually does with that line
Let’s walk one line end-to-end:

C           PRICE     ADD  TOTAL
Step 1 — raw input (80 columns)
Your compiler reads a memory buffer like:

[0..79] = 'C    ... PRICE ... ADD ... TOTAL ...'
Step 2 — extract fields (fixed columns)
RPG-II is fixed format. The compiler does:

col 6 → spec type = 'C'

cols X–Y → factor1 = "PRICE"

cols A–B → opcode = "ADD"

cols M–N → result = "TOTAL"

Step 3 — classify
spec == 'C' → calculation spec
Step 4 — opcode dispatch
opcode == ADD → build ADD IR node
Step 5 — produce IR
Something like:

IR_ADD {
  factor1 = PRICE
  result  = TOTAL
}
4. Where your macro-assembler comes in
Now we get to the important part.

The macro-assembler is NOT writing:

PRICE ADD TOTAL
It is helping you write this compiler logic cleanly.

5. Without macro-assembler (raw assembly style)
The compiler might look like:

loop:
  call read_card
  cmp r0, EOF
  beq done

  call get_col6
  cmp r1, 'C'
  bne not_c_spec

  call extract_opcode
  cmp r2, OP_ADD
  beq handle_add
  cmp r2, OP_MOVE
  beq handle_move
  b error

handle_add:
  call extract_factor1
  call extract_result
  call build_ir_add
  b loop

not_c_spec:
  ...
This quickly becomes:

branch-heavy

repetitive

error-prone

6. WITH macro-assembler (your goal)
Now the same compiler logic using structured macros:

Main loop (DO)
DO
  READ_CARD cardptr
  DOEXIT EOF

  GETCOL spec, cardptr, 6

  SELECT spec
    WHEN 'C'
      PARSE_C_SPEC cardptr
    WHEN 'O'
      PARSE_O_SPEC cardptr
    OTHERWISE
      EMIT_ERROR ERR_BAD_SPEC
  ENDSEL
ENDDO
Parsing a C-spec (SELECT + macros)
PARSE_C_SPEC MACRO cardptr
  GETSPAN opcode, \cardptr, OPC_START, OPC_END

  SELECT opcode
    WHEN OP_ADD
      BUILD_ADD \cardptr
    WHEN OP_MOVE
      BUILD_MOVE \cardptr
    OTHERWISE
      EMIT_ERROR ERR_BAD_OPCODE
  ENDSEL
MEND
Building IR (macro abstraction)
BUILD_ADD MACRO cardptr
  GETSPAN factor1, \cardptr, F1_START, F1_END
  GETSPAN result,  \cardptr, RES_START, RES_END

  IR_EMIT_ADD factor1, result
MEND
